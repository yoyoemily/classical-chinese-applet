## 工作约定

### 1. "继续项目"及板块触发词的行为

触发词规则如下：

| 用户说 | 读取哪些文件 |
|--------|-------------|
| **继续项目** | `CLAUDE.md` + `.claude/memory/MEMORY.md` + 三个板块文件（`study-section.md`、`articles-section.md`、`classics-section.md`、`mine-section.md`） |
| **继续学习板块** | `.claude/memory/study-section.md`（无需读 CLAUDE.md 和其他记忆） |
| **继续选篇板块** | `.claude/memory/articles-section.md`（无需读 CLAUDE.md 和其他记忆） |
| **继续经典板块** | `.claude/memory/classics-section.md`（无需读 CLAUDE.md 和其他记忆） |
| **继续我的板块** | `.claude/memory/mine-section.md`（无需读 CLAUDE.md 和其他记忆） |

以上触发词严格匹配，不确认、不追问，直接读取对应文件即可回答问题。**除非用户明确说"结合 git"**，否则不查 git log/diff。

**禁止操作**：
- 不要 `ls` 扫描整个项目目录
- 不要批量读取所有页面/组件文件
- 板块文件已经记录了该板块的全部信息，足够回答问题
- 不要主动读 git log、git diff、git status（除非用户明确要求结合 git）

### 2. "更新记忆"时的行为

当用户说"更新记忆"时，更新 `CLAUDE.md` 和 `.claude/memory/` 中的内容。

**记录原则：只记录结果，不记录过程**

**应该记录的**（对后续开发有帮助）：
- 新增的功能模块、页面、组件
- 新增的类型、接口、常量
- 重要的架构决策和约定
- 项目完成度的变化（哪些做完了、哪些待开发）

**不应该记录的**（过程信息，无长期价值）：
- 今天做了什么、什么时候做的
- 某个功能的实现细节、中间怎么折腾的过程
- 临时的调试信息、修复过程
- "我就是这样改的"之类的叙事

### 3. 收到图片后的行为

当用户发图片让 Claude 理解时，**禁止直接理解完图片就动手修改代码**。正确的流程是：

1. 先解读图片内容，将理解结果以文字形式反馈给用户
2. 等待用户确认理解无误
3. 确认后再动手实现

**Why:** 图片理解可能不准确，直接改代码容易跑偏，浪费时间和 token。先对齐理解，再动手。

**How to apply:**
- 看到图片 → 输出"我理解图片内容是：..." → 等用户确认 → 再改代码
- 即使图片内容看起来很明显，也要先确认

### 4. 动手前的确认原则

**始终保持一个原则：动手前一定要确认过。** 这个原则适用于所有涉及代码修改、文件操作、结构调整的场景，不只是图片理解。

**Why:** 避免返工，确保方向正确。一次错误的修改比一次多余的确认成本高得多。

**改动计划模板**

每次动手改代码之前，输出一个简短的改动计划，明确 3 点：

1. **改哪个页面/文件** —— 具体到路径，如 `pages/study/index`
2. **改哪些点** —— 逐条列出，每条一句话说清楚要做什么
3. **用什么方案** —— 每条对应一个技术方案（如"在 `init()` 中调用 `wx.setNavigationBarTitle`"）

不需要长篇大论，但必须具体。不要只说"优化样式"，要说"虚线边框 + 暖色背景 + 大号图标"。

**流程**：先输出计划 → 等用户确认（"可以"、"好的"等）→ 再动手写代码。除非修改极其简单（修 typo、改一个字面量），否则不要跳过。

**How to apply:**
- 涉及多文件修改时，先输出改动计划，等确认后再动手
- 涉及结构调整时，先确认方案
- 不确定时主动询问，不要猜测用户意图
- 单文件单处修改（如修一个变量名）可跳过，直接改

### 5. 排查问题时

如果排查问题需要扫描工程各处（如 `find`、`grep`、遍历多个目录），**必须先明确询问用户确认**，而不是默认就开始大规模扫描。

**Why:** 避免不必要的 token 消耗，保护工作上下文的缓存。CLAUDE.md + 记忆文件已经覆盖了项目 90% 的信息，大多数问题可以通过这些信息直接回答。

**How to apply:** 
- 遇到需要扫描的情况，先问："这个问题需要搜索 X、Y、Z 几个地方，我可以扫描吗？"
- 如果是单文件读取，直接操作即可，不必询问

### 6. 任务完成时明确反馈

每个任务或阶段完成后，必须明确告知用户"已完成"，不要让用户猜测是否还在进行中。完成后必须输出简短汇报，列明改动文件和需要用户手动操作的步骤。

**Why:** 用户无法判断沉默是"还在干活"还是"干完了没说"。明确的完成信号让协作顺畅，不浪费用户时间等待。

**How to apply:**
- 每阶段完成后主动汇报，格式：已完成的改动 + 需要用户手动操作的
- 不要悄无声息地结束任务

### 7. 任务完成后必须汇报

每个任务完成后，必须输出简短的汇报总结，格式为：

```
已完成的改动：
- <文件/模块>：<改动内容>
- <文件/模块>：<改动内容>

需要你手动操作的：
- <操作步骤>（如果没有则写"无"）
```

**Why:** 用户需要确认改动范围并知道下一步要做什么。不汇报就是没完成。

**How to apply:**
- 每次改完代码后立即输出汇报
- 汇报要具体到文件和改动点
- 用户说"好了？"时，如果已经汇报过就说"已汇报完毕"，如果没汇报就补一个

### 8. 前端薄层原则

**核心主张**：减少视图层面的逻辑包装，让前端尽量薄。逻辑下沉到后端，前端只负责渲染。

- 后端返回的数据应"拿来即用"，前端不做 map/filter/reduce/groupBy 等二次加工
- 汇总、统计、排序、分页等逻辑由后端接口处理
- 前端只做条件渲染（`wx:if`）、样式切换、用户交互
- 前端逻辑包装前要三思（尤其是具有一定复杂度或性能损耗的），可主动提议交给后端处理

**反例**：前端拿到列表后遍历计算汇总、groupBy 分组、根据多字段组合判断生成新字段
**正例**：后端 DTO 直接返回 `summary`、`stats`、`isXxx` 等计算好的字段

### 9. 【硬性规则】JSON 文件中的中文引号

**凡写入 JSON 文件，必须严防 ASCII 双引号 `"` 出现在字符串值内部**。

中文语境下经常需要的引号用法（如引用原文、引用说法），**必须使用 Unicode 中文引号 `“` `”`（即 `“` 和 `”`）**，绝对不能使用 ASCII `"`。

**反例**（这会导致 JSON 解析失败）：
```json
{ "explanation": "孙子以"兵"指代整个军事活动" }
```

**正例**：
```json
{ "explanation": "孙子以“兵”指代整个军事活动" }
```

**Why:** JSON 规范中 `"` 是字符串定界符，值内部出现未转义的 `"` 会破坏 JSON 结构。中文引号 `“”` 是独立的 Unicode 字符，与 JSON 语法完全不冲突。

**How to apply:**
- 写 JSON 之前，先在脑中过一遍：这个文件的每个字符串值里有没有需要引用原文的地方——如果有，全部用 `“”`
- 写完 JSON 后，读回文件内容，grep 检查是否有多余的 ASCII `"` 出现在不该出现的位置
- 如果 JSON 文件较大（>10KB），**写入前先用 Python `json.loads()` 验证**，发现解析错误立即修正再写

### 10. 【硬性规则】JSON文件修改先备份

**凡是修改超过 30KB 的JSON文件，必须先复制副本**。

```bash
cp <target_file> <target_file>.bak
```

任务完成后确认文件正确，再删除备份：

```bash
rm <target_file>.bak
```

**Why:** JSON文件修改容易因多次编辑逐步损坏内容，一旦中间某步改坏，原内容就永久丢失。有备份就可以随时恢复重来。

**How to apply:**
- 修改前 `cp file file.bak`
- 每步编辑后验证（JSON 用 `python3 -c "import json; json.load(open('file'))"` 快速校验）
- 全部完成后确认无误，`rm file.bak`
- `*.bak` 应在 `.gitignore` 中（已有配置）
